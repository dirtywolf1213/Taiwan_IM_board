# Codex 專用協作指令

> 本文件 **只給 Codex 使用**。本 repo 也會有 Claude、ChatGPT、Gemini 或其他 AI 協作；其他 AI 應以 `docs/` 與 `review/README.md` 為準，不必套用本文件。Codex 在本 repo 工作時，先讀本文件，再依任務補讀相關 repo 文件。

## Repo 定位

這是台灣內科專科醫師甄審考古題刷題網站。詳解的目的不是配答案，也不是快速複習卡，而是產出可供住院醫師準備 board / 專科考使用的高品質草稿。

詳解必須：

- 以繁體中文為主，專業名詞、藥名、trial、score、guideline 名稱保留英文。
- 寫到專科考深度：機轉、流行病學、鑑別推理、數值 / cutoff、分情境決策、常見陷阱都要說清楚。
- 能被醫師審核、能追溯來源、能直接放進 `src/data/explanations.<年>.json`。
- 永遠先標 `draft`；Codex 不可自行把任何詳解改成 `reviewed`。

## Codex 每次開工前必讀

若任務涉及詳解撰寫、審核、改寫、入庫，至少讀：

1. `docs/詳解撰寫規範.md`
2. `docs/AI寫詳解教學與指令.md`
3. `docs/AI_REVIEWER_README.md`
4. `review/README.md`
5. 相關年度的 `src/data/questions.<年>.json`
6. 相關年度的 `src/data/explanations.<年>.json`

若涉及科目範圍或題號分布，再讀 `docs/題目分類說明.md`。

## 詳解固定格式

每題 `text` 必須依序使用五段式 Markdown 標題：

```md
### 本題觀念
### 選項分析
### 答案解析
### 核心知識點
### 參考資料
```

`status` 一律使用：

```json
{ "text": "<Markdown 詳解>", "status": "draft" }
```

除非使用者明確提供人工審核決定，並要求執行 review apply 流程，否則不可改成 `reviewed`。

## 絕對避免 112 年那種空殼詳解

不要寫這類罐頭句：

- 「此選項不是本題指定答案」
- 「常見陷阱是只抓單一症狀或單一檢驗值」
- 「此選項最能符合題幹關鍵線索」
- 「正解依考題給定答案為...」
- 只把題幹重貼一遍，沒有真正推理

錯誤示範的問題是：看似有五段，但沒有醫學內容、沒有逐項推理、沒有說明選項為什麼對或錯。這種內容不得入庫作為新詳解。

目標品質請對齊 `src/data/explanations.114.json` 的 `114-087`：要能從題幹資料推到診斷 / 處置，並說清楚每個選項的病理生理、臨床情境與陷阱。

## 選項分析要求

每個選項 A-E 都要分析，且不能只說「不是答案」。

每個選項應包含：

- 正確、錯誤、部分正確、過時、或情境依賴。
- 為什麼對 / 錯：機轉、診斷標準、治療邏輯、cutoff、禁忌或 guideline 分級。
- 為什麼它是誘答：和哪個疾病、藥物、數值、病原、影像或治療時機混淆。
- 若有教學價值，說明「題幹若改成什麼情境，這個選項才會變正確」。
- 正解標示 `**正確 → 本題答案**`；若題目問「何者錯誤 / 何者不適當 / 除外」，要明確寫出這是被選出的錯誤選項。

## 文獻與查證

醫學主張必須有來源。Codex 不得憑記憶捏造 PMID、DOI、trial 結果、guideline 年份、劑量或 cutoff。

撰寫每題時：

- 必須查最新版 guideline；優先 2023-2026。
- 若權威指引確實未改版，要標註「現行版，尚未改版」。
- 每題通常列 2-4 筆參考資料，至少包含最新版指引與一篇關鍵 trial / systematic review；若主題不適合 trial，可用權威 guideline、政府 / 學會文件或標準教科書補足。
- 台灣在地議題，如健保給付、疫苗政策、法定傳染病、台灣流行病學，要查台灣來源：疾管署、衛福部、健保署、台灣專科醫學會等。
- 考試答案依該考年標準判斷；若最新指引已改變，用「最新進展」補充，不可直接把考古題正解改成新版邏輯。

參考資料需盡量完整：

- 期刊 / 指引：作者、標題、期刊、年、卷期、頁碼、PMID、DOI 或官方網址。
- 教科書：書名、版次、出版社、出版年、章節。
- 政府 / 學會文件：發行單位、標題、年份 / 版本、官方網址。

## 審核角色

若任務是「審核詳解」，Codex 要切換成獨立 reviewer 心態：

- 預設懷疑，不做橡皮圖章。
- 不確定就 `FLAG`，不要硬判 OK。
- 優先檢查：答案方向、是否問錯誤 / 除外、劑量、cutoff、病原特性、引用是否對得上、PMID / DOI 是否真實、是否漏掉新版 guideline。
- AI reviewer 只產生意見；不可自行定稿或 promote 為 `reviewed`。

審核輸出格式依 `docs/AI_REVIEWER_README.md`：

```text
<id> | OK 或 FLAG | 疑慮 | 建議修正
```

## 入庫與驗證

詳解檔位置：

- `src/data/explanations.<年>.json`

題庫檔位置：

- `src/data/questions.<年>.json`

工作原則：

- 詳解與題庫解耦，不要把詳解寫回 `questions.<年>.json`。
- 合併 JSON 時保留既有題目，不要覆蓋整個檔案造成其他題消失。
- 修改後執行驗證：

```bash
python3 tools/validate.py
```

若使用 npm workflow，也可執行：

```bash
npm run validate
```

## Codex 回報使用者時

回報要簡潔但具體：

- 說明改了哪些年度 / 科目 / 題號。
- 說明是否仍為 `draft`。
- 說明是否跑過 `tools/validate.py` 或 `npm run validate`。
- 若有未查到、來源衝突、考年標準與最新版不同，要明講，不要藏。

## 核心紅線

- 不寫空泛罐頭詳解。
- 不跳過任何選項。
- 不杜撰文獻、PMID、DOI、劑量、cutoff。
- 不把 AI 草稿標成 `reviewed`。
- 不用最新版醫學硬改考古題官方答案。
- 不引用低品質來源來支撐高風險醫學主張。
- 不為了看起來完整而寫自己沒有把握的內容。
