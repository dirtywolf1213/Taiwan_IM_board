Lesson0

想清楚:你到底要做什麼

所有專案的第一步不是寫程式,是「定義問題」。

🎯 這一步要做什麼
用一段話講清楚:給誰用、解決什麼痛點、規模多大、有什麼限制。這決定後面所有技術選擇。

💡 觀念:先回答五個問題

問題為什麼重要
① 使用對象是誰?自用?小團體?還是公開大眾?→ 影響要不要帳號、要不要伺服器。
② 內容規模多大?幾十題 vs 幾千題 → 影響資料要不要分檔、要不要「按需載入」。
③ 內容多久更新一次?很少更新 → 可「打包進網頁」;天天變 → 才需要資料庫。
④ 要不要跨裝置同步?要同步就需要後端+帳號;不要的話可以極簡。
⑤ 預算/維運能力?沒預算、不想維運 → 往「純前端、免費主機」走。
📚 教案:我們的答案
  • 對象:準備內科專科甄審的醫師,自用 + 小團體、不公開。
  • 規模:104–114 年 2000 題、11 科(每年 160 或 200 題)。
  • 更新:題庫一年加一次,平常不太動 → 適合「打包進網頁」。
  • 同步:不需要雲端同步,各自在自己裝置刷即可。
  • 預算:零維運成本是硬需求。
2000題

$0維運
🔁 換你思考
把你想做的東西,用上面五題各回答一句。如果你的答案是「自用、內容固定、不用同步、零預算」——恭喜,你會走到跟我們一樣的「純前端」路線(下一課)。如果是「要開放註冊、內容每天變」,那就需要後端,屬另一條路。
Lesson1

技術選型:要不要「後端」?用什麼蓋?

這是最關鍵的分岔路。選對了,後面省一半力氣。

🎯 這一步要做什麼
決定網站的「型態」(有沒有後端)與「建材」(用什麼框架做)。

💡 觀念一:前端 vs 後端,差在哪

前端後端
是什麼使用者眼睛看得到、手點得到的畫面躲在背後的伺服器 + 資料庫
負責顯示題目、互動、存本機進度帳號、跨裝置資料、即時運算
成本可放免費主機、不用維運要租伺服器、要維護、會當機
⚖️ 取捨
純前端:零維運、超穩、隱私好,但不能跨裝置同步、內容更新要重新打包上線。有後端:能同步能即時改,但要錢要顧。內容固定的刷題庫,純前端幾乎完勝。
📖 名詞
純前端=整個網站像「會互動的電子書」,題目全包在裡面,沒有中央伺服器在算東西。

💡 觀念二:用什麼「建材與工具」

純前端網站常見組合:

工具角色(白話)
HTML/CSS/JavaScript網頁的三大基本元素:結構 / 外觀 / 互動。
框架(React 等)把畫面拆成可重複用的「積木(元件)」,複雜畫面好維護。
打包工具(Vite 等)把程式碼「編譯」成瀏覽器看得懂的上線成品;開發時還能即時預覽。
PWA 能力讓網頁「可安裝、可離線」(Lesson 10 細談)。
📚 教案:我們的選擇
  • 純前端、零後端、不需帳號:進度只存瀏覽器本機(下一段名詞)。
  • React 19 + Vite 7:用 React 把畫面拆成 15 塊元件;Vite 負責打包與開發預覽。
  • PWA:可加到主畫面、可離線刷題。
  • 樣式用單一 styles.css(沒有額外 UI 套件,保持輕量)。
📖 名詞:localStorage
瀏覽器自帶的一小塊「私人記事本」,網站可以把你的作答紀錄存在這裡。換瀏覽器/清資料就會消失——這是純前端「不用後端也能記住進度」的關鍵,代價是不能自動跨裝置。
🔁 換你思考
不想學框架?最小可行版本甚至能用「一個 HTML 檔 + 一段 JavaScript + 題目陣列」做出來。框架是「題量大、功能多」時才划算的投資。先想清楚你要多少功能,再決定要不要上 React。
Lesson2

把內容數位化:從「來源」到「電腦看得懂的資料」

有網站殼還不夠,真正的價值是內容。先把題目變成資料。

🎯 這一步要做什麼
把題目從原始來源(PDF、Word、紙本…)變成電腦能精準讀取的「結構化資料」。

💡 觀念:電腦不懂「一份考卷」,只懂「欄位」

PDF 在電腦眼中只是「一堆圖文」,它不知道哪段是題幹、哪段是選項、答案是什麼。你必須把它拆成固定欄位——這個過程叫「結構化」。常見做法:程式自動抽取 + 人工/AI 校對。

📖 名詞:JSON
一種「結構化資料表」的檔案格式。就像把病歷拆成固定欄位(姓名、年齡、診斷…),電腦能精準讀。我們的每一題就是一筆 JSON 紀錄。
📚 教案:PDF → JSON 的半自動流程
  1. 原始 PDF 放 source-pdfs/<年>/。
  2. 寫一支轉檔小程式 tools/parse_pdf.py,把文字抽出來、依題號切開。
  3. 因為不同年份排版不同,做了多個版本(parse_pdf.py / parse_pdf_109.py / parse_pdf_old.py),像不同年代病歷要用不同判讀規則。
  4. 輸出成每年一檔:src/data/questions.114.json …
🚨 最高原則:零錯誤
醫療內容「寧缺勿錯」——抽出來的題目/答案必須與來源逐字一致,所以是「程式抽 + 人/AI 校對」,寧慢求準。
🔁 換你思考
如果你的題目本來就在 Excel 或 Word,數位化會輕鬆很多(可直接轉成 JSON)。PDF 是最麻煩的來源。先盤點你的內容「原本長什麼樣」,難度差很多。
Lesson3

設計「一題」要有哪些欄位

資料結構設計得好,後面的篩選、搜尋、統計才做得出來。

🎯 這一步要做什麼
決定「一題」要記錄哪些資訊(欄位),以及為什麼。

💡 觀念:先想「使用者會怎麼用」,再回推欄位

你想讓使用者「依年份練、依科目練、搜尋、看正確率」嗎?那每題就得有「年份、科目、答案」等欄位。功能需求決定欄位設計。

📚 教案:我們一題的長相
// questions.114.json 裡的一題
{
  "id": "114-001",        // 全站唯一編號(114年第1題)→ 才能精準定位每一題
  "year": 114, "num": 1,    // 年份、題號 → 依年份練習、原卷排序
  "subject": "心臟血管",    // 科目 → 依科目練習、各科正確率
  "question": "…題幹…",
  "options": ["A","B","C","D","E"],  // 五個選項
  "answer": 2,              // 正解索引(0起算 → 2=第3個=C)
  "answerLetter": "C",     // 正解字母(驗證會檢查兩者一致,防打錯)
  "needsImage": false,      // 有沒有附圖
  "image": "",             // 圖片路徑(有圖才填)
  "topic": "HF"           // 考點(下一課再加上去)
}

設計巧思:同時存 answer(數字)和 answerLetter(字母),程式會自動檢查兩者一致——這是一道「防呆」。送分/複選題則允許答案寫成 "A、C(送分)" 這種字串。

🔁 換你思考
畫一張你自己的「一題卡」:你要哪些欄位?要不要難度?要不要標籤?寧可一開始多想幾個欄位(像 topic 我們就是後來才加),改資料結構比改程式麻煩。
Lesson4

圖片題怎麼處理(ECG / X 光 / 病理)

內科題很多要看圖,圖要「切對、放對、對到正確題號」。

🎯 這一步要做什麼
把附圖從來源切出來、存成圖檔、對應到正確的題,並讓網頁顯示得出來。

💡 觀念:圖是獨立檔案,用「路徑」連到題

圖片不會塞進 JSON,而是存成獨立圖檔,題目裡只記「圖在哪」(路徑)。網頁顯示時再去把圖載進來。

📚 教案:自動裁切 + 回填路徑
  1. 附圖 PDF 放 source-pdfs/<年>/,上面標「第 N 題」。
  2. 工具 tools/extract_figures.py 依事先量好的「裁切框座標」切圖,存成 public/images/<年>/<題號>.jpg,並自動把路徑回填到那題的 image 欄。
  3. 顯示時自動加上正確網址前綴。
✅ 品質結果
目前 181 題有圖,經逐張開圖核對:無放錯、無重複、無左右顛倒;還因此抓到並修正了 3 題詳解的判讀錯誤(例如把皮肌炎誤判成硬皮症)。
🔁 換你思考
如果你的圖本來就一張一張分開(不是擠在一張 PDF),就不用裁切,直接放進資料夾、填路徑即可。圖片題最花工的是「裁切」與「對題號」,先評估你的圖長怎樣。
Lesson5

分類與索引:讓題庫「好找、好統計」

2000 題不分類就是一團亂。分類 + 做目錄,功能才強。

🎯 這一步要做什麼
幫每題分科、分考點,並做一份「輕量目錄」讓網站跑得快。

💡 觀念一:兩層分類(科 → 考點)

大分類(科)讓你選「心臟科」;細分類(考點)讓你只練「ACS」或「心衰竭」。考點用英文縮寫(符合醫界習慣)。分類可用「關鍵字規則」自動做,再人工微調。

📚 教案:分科 + 分考點 + 細分
  • tools/classify.py 把題分到 11 科。
  • tools/classify_topic.py 依「題幹+選項+詳解」關鍵字標考點(如 ACS/HF/SLE/Gout…),2000 題全覆蓋。
  • 大桶還能再細分:血液腫瘤的實體癌拆成 Onc-Lung/Breast/CRC/GU/GI… 等器官別。

💡 觀念二:輕量索引(目錄),讓網站不卡

首頁要顯示各科題數、正確率,搜尋要快。若每次都載入全部 2000 題的完整內容會很慢。解法:做一份只含「id/年/題號/科目/考點」的薄目錄,先讀它;真要做某年的題,才載入那一年。

📚 教案:index + 按需載入
tools/build_index.py 產生 src/data/index.json(薄目錄)。App 啟動先讀目錄,點某年才動態載入該年題目——像先看病歷封面索引,要看某次住院才抽那一本。
🔁 換你思考
題量小(幾百題)其實可以全部一次載入,不必做索引。索引是「題量大 + 要即時統計/搜尋」時的優化。先別過度設計。
Lesson6

詳解系統:解耦、五段式、草稿制

刷題網站的靈魂。詳解要好寫、好維護、可信任。

🎯 這一步要做什麼
設計「詳解怎麼存、長什麼格式、怎麼標示可信度」。

💡 觀念一:詳解與題目「分開存」(解耦)

把詳解放在另一個檔(以題號對應),好處是:重跑題目轉檔時不會蓋掉辛苦寫的詳解;詳解可以慢慢補。

📚 教案:詳解獨立檔 + 狀態
// explanations.114.json
{
  "114-001": {
    "text": "### 本題觀念\n…(Markdown 五段式)",
    "status": "draft"   // draft=草稿;reviewed=醫師審核定稿
  }
}
📖 名詞:Markdown
用簡單符號排版的純文字。打 **字** 變粗體、### 標題 變小標、[文字](網址) 變連結。詳解就用它寫,網頁會自動轉成漂亮排版。

💡 觀念二:固定格式 + 可信度標示

固定「五段式」讓每篇詳解品質一致、好讀;用「草稿/已審核」標示讓使用者知道能信到什麼程度。

五段式內容
### 本題觀念這題在考什麼核心概念
### 選項分析逐一說明每個選項對錯與陷阱
### 答案解析為什麼選這個、與正解一致
### 核心知識點可帶走的重點(藥名劑量、cutoff、分情境決策)
### 參考資料最新版指引 + 關鍵試驗,附可點的 PMID/DOI
✅ 草稿制(信任設計)
所有 AI 起草的詳解一律先標 draft,網頁顯示「草稿・未經審核」;只有醫師人工定稿才改 reviewed。AI 永遠不可自行標 reviewed。目前 2000 篇:17 已審、1983 草稿。
⚠️ 正解依「考年標準」
確保答對當年題;考點後來更新,用「> 最新進展:」另外註明,但不改考年正解。
🔁 換你思考
你的詳解要不要固定格式?強烈建議要——格式一致可讀性與品質都會大幅提升,也方便日後請 AI 大量起草。先訂一份「撰寫規範」。
Lesson7

把畫面做出來(積木式元件 + 各種練習模式)

資料齊了,現在讓使用者真的能「刷」。

🎯 這一步要做什麼
設計使用者流程與畫面:首頁 → 選模式 → 一題一題作答 → 看詳解。

💡 觀念:把畫面拆成可重複用的「積木」

「一題的卡片」會在練習、模擬考、搜尋結果裡重複出現——做成一塊積木(元件)就能到處用,改一次到處更新。

📖 名詞:元件(component)
畫面的積木。例如 QuestionView(一題卡)、Home(首頁)、Search(搜尋頁)各是一塊,可組合、可重用。
📚 教案:15 塊元件 + 多種模式

畫面在 src/components/,由 App.jsx 當「總機」切換。使用者能用的模式:

模式/功能說明
依年份/依科目/隨機 練習依科目時上方有考點 chips,可只練某考點
到期複習(間隔複習)用 Leitner 演算法排程:答對隔越久才再出現,記最牢
錯題複習 / 我的收藏 ★只練答錯過的 / 收藏的
模擬考可限時(每題1分鐘、到時自動交卷)、自訂題數;交卷給各科+各考點弱點報告
搜尋 🔍搜題幹/選項/題號/科目/考點,關鍵字高亮
跳題格永久標記已作答(綠=對、紅=錯)
學習統計各科/各年完成度與正確率,可匯出 PDF 報告
收藏★/筆記📝/深色模式/鍵盤快捷鍵/單題分享連結各種體驗優化
🔁 換你思考
第一版別貪心。最小可行版只要:列題 → 點選項 → 顯示對錯與詳解。其他模式(模擬考、SRS、統計)都是後續一個一個加上去的(我們也是這樣長出來的)。
Lesson8

記住使用者的進度(沒有後端也能做到)

作答紀錄、收藏、筆記要存下來,而且不靠伺服器。

🎯 這一步要做什麼
把使用者的作答對錯、收藏、筆記、複習排程存起來,下次打開還在。

💡 觀念:存在「瀏覽器本機」

純前端沒有資料庫,但瀏覽器自帶 localStorage(本機記事本),可以把進度存在使用者自己的裝置。優點:不用後端、隱私好;缺點:換裝置不會跟著走 → 要做「備份/還原」。

📚 教案:本機進度 + 備份 + 間隔複習
  • 進度存在 localStorage,內容包含:每題最後一次對錯、收藏清單、筆記、間隔複習排程。
  • 備份:「匯出進度」成備份碼或檔案(還做了壓縮,2000 題的紀錄也很短);換裝置「匯入」還原。
  • 間隔複習(SRS):用 Leitner 盒子演算法決定每題「下次該何時再出現」。
⚠️ 提醒使用者備份
因為只存本機,換手機/清快取前要先匯出。所以「按更新前會跳出提醒先備份」。
🔁 換你思考
哪天你真的需要「跨裝置同步」或「老師看學生進度」,那才是你需要後端+帳號的時機。在那之前,localStorage + 備份碼就很夠用。
Lesson9

品質把關:讓錯誤無所遁形

醫療內容最怕錯。用「自動檢查 + 原則」雙重把關。

🎯 這一步要做什麼
建立「每次改完都自動檢查一遍」的機制,並訂下不可違反的內容原則。

💡 觀念:能自動檢查的,就別靠人記得

寫一支「驗證程式」,把所有「機器可判斷的錯」一次抓出來(欄位漏填、答案不一致、圖檔不存在…),人就能專心看「機器看不出的醫學正確性」。

📚 教案:自動驗證 + 人為原則

tools/validate.py 會檢查:必要欄位、選項=5、答案字母與索引一致、科目合法、題號連續不重複、圖檔存在、目錄同步、詳解覆蓋率。另有人為鐵則:

  • 參考資料 PMID/DOI 必須查證為真(查不到改原則性描述,嚴禁杜撰)。
  • 圖片題逐張開圖核對(放對沒、判讀對沒)。
🚨 最高原則:零錯誤優先
不確定的數字/試驗/PMID 一律查證或改原則性描述。寧缺勿錯。
🔁 換你思考
列出你內容裡「一定不能錯、又能用規則檢查」的點(例如答案格式、必填欄位),把它寫成檢查清單——這份清單就是你的「validate」。每次改完跑一次,睡得安穩。
Lesson10

變成可安裝、可離線的 App(PWA)

讓網頁能「加到主畫面」、沒網路也能用。

🎯 這一步要做什麼
讓你的網站能像 App 一樣安裝、離線使用,並能優雅地提示更新。
📖 名詞:PWA
Progressive Web App——「漸進式網頁應用」。簡單說:用網址開的網頁,但能加到主畫面、離線運作,介於網站與 App 之間。背後靠一個叫 Service Worker 的小幫手把檔案快取(暫存)在裝置上。
📚 教案:提示式更新
  • 題目與圖第一次看過後會被快取,之後可離線刷。
  • 提示式更新:偵測到新版時,畫面下方跳「🆕 有新版本可用」,使用者按「立即更新」才套用——不背景強制換版。
  • 每 30 分鐘、切回分頁時會自動檢查一次更新。
🔁 換你思考
PWA 是「加分項」,不是必需。第一版可以先不做;等你發現使用者想「裝起來、離線用」時再加。打包工具(Vite)有現成外掛,加起來不難。
Lesson11

打包與上線(讓全世界連得到)

在自己電腦能跑還不算數,要「打包」並「部署」到主機。

🎯 這一步要做什麼
把程式碼變成上線成品,並放到網路主機上,拿到一個網址。

💡 觀念:build(打包)→ deploy(部署)

build把程式碼「編譯」成瀏覽器看得懂的成品檔案。像把食譜做成上桌的菜。
deploy把成品放到網路主機上,別人才連得到。像把菜端到餐廳給客人。
📚 教案:兩個免費主機同時上線
主機說明觸發
GitHub Pages放在子網址 /Taiwan_IM_board/程式碼推上 main 主線就自動打包上線
Cloudflare Worker放在根網域(自己的網址)用 npx wrangler deploy 部署

兩者唯一要小心的差別是「網址前綴」設定(子網址 vs 根網址),設錯會整頁空白——這在設定檔已處理好。常用指令:npm run build(打包)、npm run dev(本機預覽)。

🔁 換你思考
免費主機很多(GitHub Pages、Cloudflare、Netlify、Vercel…),純前端網站幾乎都能免費掛上去。先挑一個就好,熟了再考慮第二個當備援。
Lesson12

上線後的維護:版本、更新、安全網

網站會一直長大,要有「記錄改了什麼」和「能還原」的機制。

🎯 這一步要做什麼
建立「版本紀錄」「更新提示」與「可追溯/可還原」的維護習慣。
📖 名詞:Git
「版本控制」工具,等於專案的完整修改病歷 + 時光機。commit=存一次檔並附說明;branch=另開實驗支線不動正式版;merge=把支線併回主線;main=正式版支線。
📚 教案:維護的三個習慣
  • 版本紀錄:changelog.json 記每次更新(第一筆的版本號就是 App 顯示的版本,目前 v1.8.2)。
  • 「改功能 = 三件事一起改」:任何使用者看得到的更動,必須同步更新 ① changelog ② App 使用說明 ③ README,確保「程式、說明、版本」永遠對得上。
  • Git 工作流:在功能分支開發 → 驗證 → 合併回 main(自動上線);每次 commit 都留清楚說明,任何更動都查得到、能還原。
🔁 換你思考
就算只有你一個人做,也務必用 Git。它是你的後悔藥:改壞了能回到前一版、想知道某段什麼時候改的也查得到。對醫療內容尤其重要。
Lesson13

進階:用 AI 大規模協作(2000 題怎麼審完)

內容量大時,讓多個 AI 分工,但「人做最後判斷」。

🎯 這一步要做什麼
當題量大到人工做不完時,用 AI 當產線,並設計「不會出錯」的協作流程。

💡 觀念:AI 做粗活,人做判斷;而且要「雙重查證」

AI 很會大量起草與比對,但會犯錯。關鍵設計是:讓不同 AI 互相查核,且重要事實(如 PMID)每個環節都各自再查一次,人保留最終決定權。

❓ 常見問題

完全不會寫程式,真的能做出來嗎?

能。理解「整件事怎麼運作、每步在決定什麼」最重要(這份教學的目的);實作層面現在可以大量交給 AI 協助,你負責內容與把關。

純前端,題目放哪、會被亂改嗎?

題目打包在網頁成品裡放在主機上。任何更動都要透過 Git 留紀錄再上線,可追溯可還原;一般使用者只能讀。

為什麼不做帳號/雲端同步,不是比較方便?

那需要後端+維運+成本+資安責任。對「自用/小團體、內容固定」的刷題庫,純前端+備份碼的性價比更高。等真的需要再升級。

「草稿・未經審核」能信嗎?

表示 AI 起草、尚未醫師定稿。已經過 AI 互審與 PMID 查證,但請以專業判斷為準、自行查證。醫師審核後會拿掉標記。
